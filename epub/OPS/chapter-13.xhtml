4章：ブログ記事

私は数年間ブログを書いており、デザインの見直しを4、5回行いました。多くのデザイナーブロガーと同じように、あちこちいじらずにはいられず、今も結果には満足していません。しかし、時間は貴重なものです。特にクライアントから依頼されたプロジェクトでは──時にはこの理由で、作業が途中で打ち切られることもあるでしょう。

個人的な意見ですが、最も効果的なブログのデザインは、コンテンツを中心とし、じゃまなものはできる限り取り除いた質素なデザインです。パフォーマンスとアクセシビリティのどちらの点でも、簡素であることには計り知れないほどの効果があります。複雑で凝ったレイアウトを作り込まないことも、より堅牢で柔軟な閲覧体験につながります。読者が何らかの調整を行った場合でも、破綻することが少なくなります。

ここでは、インクルーシブなブログ記事の構成について探りましょう。ブログ記事はパーマリンクで見ることができ、親ページの主要なコンテンツを構成する要素でもあります。セマンティックな構造とソースの順序を考慮に入れなければなりません。また、効果的なリンクテキスト、ビデオコンテンツの組み込み、コンテンツ編集者の作業を容易にする自動化されたフローシステムの作成についても説明します。

訳註

パーマリンク (permalink) とはブログの個々の記事を見るためにつけられたURLや、そのURLが指定されたリンクのことを指しますが、ここではそのリンク先で見られる個別記事ページの事を指しています。

<main> 要素

2章「ドキュメント全体」で設定した<main>要素を思い出してください。すべてのWebページには、そのページ独自のコンテンツを特定する<main>要素があるべきです。この要素は、そのページの実際のメインコンテンツは何であるかについて考えるのに役立ちます。また、後でスクリーンリーダーユーザーがナビゲーション手段として利用することもできます。

ブログのパーマリンクでは、主要なコンテンツがブログ記事そのものであるため、これを<main>の中に置く必要があります。下のidに注意してください。これにより、キーボードユーザーがスキップリンクを介して記事に移動できるようになります（詳しくは2章「ドキュメント全体」で説明しています）。

<main id="main">
    <!-- ここにブログ記事 -->
</main>

複雑なグリッドシステムを組み込むとき、私たちは<div>ベースの枠組みに大きく依存しがちです。これはページの視覚的な構造をくくるのに役立つかもしれませんが、セマンティックな構造にはなんの効果もありません。つまり、以下のコード例は、セマンティック的には前のコードと同じです。ただソースコードが複雑になっているにすぎません。

<div class="grid_50">
    <div class="grid_50_inner">
        <main id="main">
            <div class="main_inner">
                <!-- ここにブログ記事 -->
            </div>
        </main>
    </div>
</div>

一般的なケースでは、サードパーティのグリッドシステムを使うことはおすすめできません。通常、それ自体が大きなアセットとなり、複雑で冗長なマークアップが増加します。ダウンロードするドキュメントが大きくなるだけでなく、やっかいな副作用が発生することもあります。たとえば、私がテストをしていたときに、スクリーンリーダーが複雑なDOMの解析に苦労し、誤った出力をしたことがありました。

非常に複雑なレイアウトを構築するのであれば、グリッドシステムは役に立つかもしれません。しかし、非常に複雑なレイアウトは構築すべきでないのです。シンプルなユーザーインターフェイスこそが、ユーザブルなインターフェイスであり、アクセシブルなインターフェイスであり、メンテナブルなインターフェイスであり、ハイパフォーマンスなインターフェイスなのです。

印刷デザインでは、グリッドに沿った要素の相対位置には信頼性と一貫性があるため、意味を伝えるために横に並べて配置したり、「右の写真」といった言い方をしたりできます。レスポンシブWebには、このような贅沢はできません。時には、空きスペースを徹底的に埋めるために、ものを横並びにしたいと思うことがあるでしょう。しかし、厳密にやろうとするのは無駄なことです。Flexboxを使用して、 レスポンシブな自己管理型のグリッドを作る簡単な方法については、10章「フィルターウィジェット」で詳しく説明します。

見出しの構造

HTMLのセマンティクスは、意味を提供する個々の要素の観点だけで見られてしまいがちで、他の要素との関係は考慮されない傾向にあります。<main>要素そのものが独特な動作を提供しているのは確かですが、これらのメリットは、不適切な配置によって損なわれることもあります。たとえば、単一のページで複数の<main>要素を使ったり、<main>を使用してDOMツリーの大半をラップしたりする使い方です。

訳註

ひとつの文書に複数のmainを置くことの是非については、複数の立場があります。原著が出版された2016年10月の時点では、W3CのHTML5.0勧告には、ひとつのドキュメント内で複数のmain要素を使ってはならないという趣旨の記述がありました。しかし、その後のHTML5.1では該当部分が削除され、HTML5.2では「ひとつのドキュメントに可視のmain要素を複数含めてはならない」という趣旨に変更になっています。また、WAI-ARIA1.1では「コンテンツ制作者はdocumentやapplicationの中で複数の要素にmainロールをマークづけすべきではない」という主旨のことが書かれています。

なお、WHATWGのHTML Living Standardには、複数mainを禁止する旨の記述はありません。

参考

  • HTML5 W3C Recommendation 28 October 2014 - 4.4.14 The main element
  • HTML 5.1 2nd Edition W3C Recommendation 3 October 2017 - 4.4.14 The main element
  • HTML 5.2 W3C Candidate Recommendation, 8 August 2017 - 4.4.14 The main element
  • Accessible Rich Internet Applications (WAI-ARIA) 1.1 - main (role)

特に見出し要素は、意味の多くが互いの関係性からもたらされます。

<h1>要素

<h1>要素は、ドキュメントの第1レベルの見出しを表します。そのドキュメントの先頭の見出しです。ここでのパーマリンクでは、ブログ記事のタイトルそのものにします。<h1>テキストは実質的に、ドキュメントの本文である最上位セクションのラベルとなります。このため、1ページに複数あるのは論理的に正しくありません。

<main id="main">
    <h1>インクルーシブなブログ記事のマークアップ方法</h1>
</main>

訳註

HTML5のセクショニングアルゴリズムでは、bodyもセクションを構成します。そのため、bodyに対する見出しがない状態でsection要素を使うと、最上位セクションに見出しが付けられていない状態になります。たとえば以下のようなマークアップがあるとします。

<body>
<section>
<h1>最初の見出し</h1>
<p>本文</p>
</section>
<section>
<h1>次の見出し</h1>
<p>本文</p>
</section>
</body>

このとき、アウトラインは以下のようになり、最上位に見出しのないセクションができてしまいます。

1.(untitled section)
1-1.最初の見出し
1-2.次の見出し

なお、明示的セクションを使わない場合は、h1要素を複数使うとそのままbodyが分割されます。

HTML Living Standardには "A document can contain multiple top-level headings:" として以下のようなサンプルが掲載されています。

<!DOCTYPE HTML>
<html lang=en>
<title>Alphabetic Fruit</title>
<h1>Apples</h1>
<p>Pomaceous.</p>
<h1>Bananas</h1>
<p>Edible.</p>
<h1>Carambola</h1>
<p>Star.</p>

このときのアウトラインは以下のようになり、見出しのないセクションは現れません。つまり2017年現在の仕様では、書き方によっては複数のh1要素を使うこともできるということです。

1. Apples
2. Bananas
3. Carambola
参考
  • HTML Living Standard - 4.3.11.2 Sample outlines

<main>要素と同じく、<h1>は、支援技術のユーザーにナビゲーション用のショートカットを提供します。NVDAやJAWSでは、1キーを押すと直接<h1>に移動でき、「インクルーシブなブログ記事のマークアップ方法、見出しレベル1」と読み上げられます。

実質的に、<main>と<h1> は似たような機能を提供していますが、両方を採用するのはなぜでしょうか？これは、スクリーンリーダーユーザーがみな同じではないからです。私が行った、スクリーンリーダーユーザーの行動および好みに関する非公式の研究では、一部の回答者はランドマーク（リージョン）ナビゲーションを使いこなしていたものの、大半の回答者は見出しを使って移動していました。ユーザーに目立たない複数のオプションを提供することを、マルチモダリティと呼ぶことがあります。それこそが、ビデオコンテンツにトランスクリプト（内容を文字に書き起こしたもの）を添えるべき理由です。これについては、この後、詳しく説明します。

訳註

ユーザーインターフェイス論における「マルチモダリティ」という言葉は、一般的には、複数の異なる感覚を用いて入出力ができることを指します。たとえば、目で見るだけでなく音で聞くこともできる、キーボードで入力するだけでなく音声入力もできる、といった状況をマルチモダリティと呼びます。

また、回答者の何人かは、スクリーンリーダーの下矢印キー（またはそれに相当するキー）を使用して、ドキュメントを先頭から末尾に向かって、要素から要素へと移動していると答えています。ソースの順序は構造の重要な側面であり、ブログ記事は上から下に読んで意味が通らなくてはなりません。視覚的には無理やり順番を変えて見せることもできますが、このような理由から、記事の先頭を公開日ではなく、見出しで始めることが大切です。

<!-- これは使わないでください -->
<main id="main">
    <div class="meta">公開日 <time datetime="2017-12-12">2017/12/12</time></div>
    <h1>インクルーシブなブログ記事のマークアップ方法</h1>
    <!-- ここに記事のコンテンツ -->
</main>

<!-- これを使ってください -->
<main id="main">
    <h1>インクルーシブなブログ記事のマークアップ方法</h1>
    <div class="meta">公開日 <time datetime="2017-12-12">2017/12/12</time></div>
    <!-- ここに記事のコンテンツ -->
</main>

1つめの例では、汎用的なhキーを使用して見出しでナビゲートしている人には公開日付はわかりません。スクリーンリーダーは日付を読み上げずに通過してしまうからです。

なお、上記の最初のコード例の構造は、自動化されたアクセシビリティテストツールでは、おそらくエラーとはなりません。だからといって最初のコード例のほうが好ましいということではありませんので注意してください。インクルーシブデザインとは、単に技術的なエラーを修正するだけではなく、ユーザーが実際にインターフェイスをどのように使うかを評価することなのです。

サブセクション

構造に関して、ソースの順序と同じくらい重要なのが階層の深さです。ブログ記事をサブセクションに分割することで、どの箇所にどの部分が属しているかを概要やアウトラインとして表現できるようになります。HTML5仕様には<section>などのネスト可能なセクショニング要素が含まれていますが、現在のところ、インクルーシブな方法で階層の深さを記述できる要素は <h1>～<h6>だけです。<section>や<article>は使用すべきでないとは言いませんが、これらが提供する情報では目的を果たせません。HTML5のアウトラインアルゴリズムは、セクショニング要素に基づいてセクションの構造を自動化し、<h1>～<h6>を不要にするよう設計されていました。しかし、これはどのユーザーエージェントにも実装されておらず、今後実装されそうな兆候もありません。

訳註

ユーザーエージェントをブラウザに限定しなければ、セクショニングアルゴリズムの実装は存在します。HTML5.1の勧告には、適合性チェッカーやブラウザ拡張による実装はあるものの、ブラウザネイティブの実装がないという趣旨のことが書かれています。

参考
  • HTML 5.1 - 4.3.10.1. Creating an outline

前述のように、ブログ記事は<h1>で開始すべきです。それで終わりにすることもできますが、構造情報がこれだけでは、多くのユーザーに理解の点で問題が生じます。理解しやすい塊にコンテンツを分解するよう、常に意識するべきです。HTMLの最終的な目的は、要素の識別を通じて構造化することですから。

記事内にサブセクションを作成するには、<h2>を使用します。<h2>の2が意味しているのは、「レベル2の深さ」、つまり「第1レベルから1つ下がったレベル」ということです。視覚的には、見出しはフォントサイズで区別されるのが慣習となっており、レベルが深いほど font-sizeは小さくなります。

h1 { font-size: 3em; }
h2 { font-size: 2.25em; }
h3 { font-size: 1.5em; }

ここで、多くの開発者が過ちを犯します。そのセクションの重要性を判断して、それに最も合うように思える見出し要素を選んでしまうのです。たとえば、そのセクションが重要ではないと判断し、font-sizeを小さくしたいと考えて、<h2>ではなく<h3>を選択することがあります。これは、視覚的なユーザーと非視覚的なユーザーの両方をあざむくことになります。なぜなら、知覚される構造を壊すため――レベルが1つスキップされることになるためです。

<h1>ブログ記事のマークアップ方法</h1>
<!-- 序文コンテンツ-->
<h3>「セマンティック」という単語について</h3> <!-- 待って！2番目のネストレベルはどこに？-->


ドキュメント構造内で階層が欠けていると、ドキュメントの階層を思い描こうとしているユーザーは混乱します。

CSSクラスを使うことで、見出しのスタイルを分離して処理する開発者もいます。この方法では、セマンティックな構造に依存することなく、自由にフォントサイズを変更することができます。

<h2 class="h3">私は何でしょう？</h2>

この方法では、見た目のために構造を崩さずに済みますが、注意が必要です。たいていの場合、<h2>は<h2>らしい見た目にすべきです。結局のところ、それが何であるのか、なのです！インクルージョンの役割のひとつは、多様なユーザーニーズ全体を通して、体験を等価なものにすることです。コンテンツを視覚的に扱う一方で、非視覚的には別の方法でコンテンツを処理するということは、見えているユーザーと見えていないユーザーを分離し始めているということです。RSSリーダーなどを通じて表示される際には、異なるスタイルシートが適用されるため、視覚的な構造が異なるものになることもあります。

サブタイトル

ブログ記事を書く人が1つのタイトルで書きたいことすべてを書けなかった場合、サブタイトルや小見出しを使用することがあります。これは本にも当てはまります。

前の例を書き直して、タイトルを「ブログ記事のマークアップ方法」とし、サブタイトルを「～シンプルな7つのステップで～」とすることを考えてみましょう。問題はこれをどのようにマークアップするかです。以下のように、別の見出し要素にサブタイトルを入れている実装を多数見たことがあります。

<main id="main">
    <h1>ブログ記事のマークアップ方法</h1>
    <h2> ～シンプルな7つのステップで～</h2> <!-- これは見出しにすべきではありません -->
    <div class="meta">公開日 <time datetime="2017-12-12">2017/12/12</time></div>
    <!-- ここに記事のコンテンツ -->
</main>

ここまでに見出しの構造的な役割を学習してきたみなさんには、何が問題かおわかりでしょう。それは、<h2>がすぐ下に「～シンプルな7つのステップで～」というサブセクションを作るということです。これはセクションの見出しとしては奇妙で断片的であり、以下のような、ほとんど意味を成さない目次ができることになります。

  • ブログ記事のマークアップ方法
    • ～シンプルな7つのステップで～
    • 見出し
      • サブセクション
    • リンクテキスト

一時は、<hgroup>要素が規定の解決策でした。以下のように、<h1>と<h2>を <hgroup>でラップすることで、理論上はアウトラインから<h2>を取り除き、サブ見出しが作られることを防げます。

<!-- これは使わないでください -->
<hgroup>
    <h1>ブログ記事のマークアップ方法</h1>
    <h2>～シンプルな7つのステップで～</h2>
</hgroup>

しかし、<hgroup>についてのMDN（Mozilla Developer Network）の記事に示されているように、<hgroup>はHTML5仕様から削除されています。そのため、この要素が各種のブラウザで確実に実装されていることを前提としてはなりません。

正しい解決策は、何を行いたいかによります。「～シンプルな7つのステップで～」をタイトルの一部にしたい場合は、特別な意味を持たない<span>要素でこれを区切り、h1 span{ display:block;}を使用して改行することができます。

<h1>
    ブログ記事のマークアップ方法  <span>～シンプルな7つのステップで～</span>
</h1>

こうすると以下のようなアウトラインが生成されます。

  • ブログ記事のマークアップ方法 ～シンプルな7つのステップで～
    • 見出し
      • サブセクション
    • リンクテキスト

そうではなく、サブタイトルは重要でないと考えて、アウトラインからそのテキストをまるごと削除したいこともあります。そのような場合には、以下の方法で対処できます。

<h1>ブログ記事のマークアップ方法</h1>
<p><span class="visually-hidden">サブタイトル:</span> ～シンプルな7つのステップで～</p>

.visually-hiddenクラスを指定した<span>を使っていることに注目してください。視覚的には、それがサブタイトルの役割を果たしているということは、おそらくスタイルで表現されていることでしょう。非視覚的なコンテキストでは、このことはわかりません。<h1>から<p>へと移り変わることで、スクリーンリーダーは、記事の本文を読み始めたと考えるでしょう。ここでは、「サブタイトル：」という接頭辞をつけることで、非視覚的に<p>要素の役割を明確化しています。これは、<button>要素の暗黙的なロールによって、スクリーンリーダーが「ボタン」と読み上げるのと同じようなやり方です。

この.visually-hiddenクラスには専用のCSSプロパティが定義されており、視覚的には要素を巧みに隠しつつ、スクリーンリーダーでは読み上げられるようにしています。display: none;、visibility: hidden;、height: 0;、width: 0;を使ってもうまく機能しません――視覚と聴覚の両方でspanが利用できなくなります。

.visually-hidden {
    position: absolute;
    width: 1px;
    height: 1px;
    overflow: hidden;
    clip: rect(1px, 1px, 1px, 1px);
    white-space: nowrap;
}

このようなユーティリティクラスをスタイルシートに用意し、自由に使えるようにしておくことをおすすめします。.visually-hiddenは後の章でも何度か出てきます。

もし<subtitle>要素が定義されたなら、この要素のロールが持つ機能として「サブタイトル：」という接頭辞を提供して、意味を明確にするようになるかもしれません。接頭辞は自動的に「アクセシビリティツリー」――DOMに役割(role)、プロパティ、状態(state)、ラベル、値などの情報を加え、支援技術に提供できるようにしたもの――に追加され、支援技術から利用できるようになるでしょう。セマンティックな要素が定義され、それを使うことができるようになれば、このようなメリットも生まれるのです。

<article>要素

HTML5のセクショニング要素については多くの混乱があります。たとえば、「これが記事（article）だというなら、<article>要素に入れるべきではないか？」と思われるかもしれません。

<main id="main">
    <article>
        <h1>インクルーシブなブログ記事のマークアップ方</h1>
        <div class="meta">公開日 <time datetime="2017-12-12">2017/12/12</time></div>
        <!-- ここに記事のコンテンツ -->
    </article>
</main>

その答えはこうなります。「おそらくそうではありませんが、もしかするとそうかもしれません。」理論上は、<article>があれば、HTML5のアウトラインアルゴリズムに従って、新しいサブセクションが始まるべきです。この例では、まだドキュメントのメインコンテンツ（最も外側のセクション）も開始していないため、ここに置くのは良くないでしょう。しかし、実際にはいずれのユーザーエージェントにもアウトラインアルゴリズムが実装されていないため、支援技術ユーザーにとってこれは何の効果も、害もありません。<main>要素とは異なり、<article>はランドマークではないため、ナビゲーション機能を提供しているスクリーンリーダーはほとんどありません。<main>の単一の子要素になっているため、いずれにしてもユーザーは同じ場所に移動することになります。

結局のところ、<article>の配置にはあまり魅力がなさそうですが、ある特定のコンテキストではちょっとした利点があります。スクリーンリーダーのJAWSは（そしてJAWSだけが）、この要素に入るときに「アーティクル開始」、出るときに「アーティクル終了」と読み上げます。確かに、同じページに複数の<article>があり、ユーザーが上から下に向かって要素を1つずつ読んでいる場合には、これは有益な情報です。以下のマークアップは、ブログのホームページの、最近の記事が複数リストされるところで役立ちます。

</article>
<article>

</article>

セマンティックな要素を選ぶときは、ユーザー体験の観点から考えるようにしましょう。技術的に正しい要素を使用していても、それが全くサポートされておらず、誰が見ても何の効果も発揮しないことがあります。また、効果が間違いなく発揮される場合でも、混乱を招いたり、一貫性がなかったり、じゃまだったりする場合もあります。

アリス・バートレット（Alice Bartlett）は、「Burn Your Select Tags」（ビデオ）の中で、UK Government Digital Serviceで彼女が行った<select>要素のユーザービリティの調査について話しています。この要素は広く実装されておりセマンティックでもありますが、可能な限り<select>要素の使用を避けるべき強力な根拠があります。いくつかの技術的な欠点（特定のデバイスで、オーバーレイ表示される<option>をズームできないなど）があるだけではなく、あらゆる人が理解できるわけではないという問題もあります。

プログレッシブエンハンスメントと相互運用性

ここまでにブログ記事で作り上げてきた、音として聞こえるセマンティック構造は、スクリーンリーダーユーザーにのみメリットをもたらすわけではありません。ブラウザがスタイルシートの読み込みに失敗しても、ユーザーエージェントのデフォルトスタイルシートが適用されて、インターフェイスの主要な要素は認識できるようになります。たとえば、見出しは太字テキストで表示され、セクションの深さを示すフォントサイズになります。

さらに、フィードリーダーなどのコンテンツをまとめるサービスでは、独自のスタイルを適用し、それぞれのアプリケーション内で読みやすくすることができます。さまざまなコンテキストで、さまざまな入力を介して問題なく機能するコンテンツは、相互運用性があると言うことができます。優れたセマンティック構造は、スクリーンリーダー、フィードリーダー、サーチエンジン、およびコンテンツ内の構造的な意味を探る他のあらゆるパーサーで適切に扱うことができます。

コンテンツに音としての構造があることをテストする良い方法は、CSSをオフにしてもページが読めるかどうかを確認することです。順序や階層が明確にわかりますか？ナビゲートでき、使い物になっていますか？

シングルページアプリケーションに関する注意

シングルページアプリケーションには、クライアントサイドJavaScriptを使用してコンテンツの表示と再描画を行うという特徴があります。インタラクションの結果としてコンテンツに大きな変化が生じる場合でも、このアプローチならばサーバーと何度もやり取りをせずに済み、即座に実現することができます。

しかし、コンテンツが静的で、読ませることだけが目的なら、クライアントサイドJavaScriptでコンテンツを表示するのは非効率で、比較的信頼性の低い方法です。コンテンツは、JavaScriptが無効になっているユーザーや、JavaScriptがエラーになったり、ブロックされたり、読み込めなかった場合でも利用できるようにすべきです。

また、サーバーでレンダリングされたコンテンツを提供することは、検索エンジンのユーザーがそのコンテンツを見つけられること、第三者がcURLなどのコマンドラインツールから利用できることを意味します。こうした理由から、IndieWebムーブメントの創始者のひとりであるタンテック・チェリック（Tantek Çelik）は、JavaScript依存の静的コンテンツはWebデザインの原則に則っていないと主張しています。つまりこういうことです。

「curlできなければ、Web上にはない」— タンテック・チェリック「js;dr = JavaScript required; Didn’t Read」

例えばsmashingmagazine.comなどのWebサイトは、レンダリング済みのコンテンツをクライアントに送信しているため、curlコマンドを使用することでコンテンツをコンソールに表示できます。JavaScriptによってクライアント側でコンテンツを構築しているケースでは、Webページの外枠のコードだけしか表示されません。JavaScriptのコードへのリンクも得られますが、これはcurlでは実行できません。

curl https://www.smashingmagazine.com

Flesch-Kincaid リーダビリティテスト

3章「パラグラフ」では、わかりやすい本文の書き方について基本的なガイダンスを提供しました。ブログ記事内にいくつかのパラグラフをまとめつつあるので、コンテンツの読みやすさを見直すには良いタイミングでしょう。単語、文章、パラグラフの長さを評価してテキストの読みやすさを判定するテストが2種類あります。

Flesch Reading Easeテストでは、読みやすさが0～100点で評価されます。得点が高いほど読みやすい、つまり、多様な読者に対して、よりインクルーシブであるという意味です。Flesch- Kincaid Grade Levelテストもこれと似ていますが、評価結果は米国の学校の学年として表現されます。より若い読者が読めるほど、より読みやすいという考え方です。以下は、このテストのWikipediaでの解説ページからの要約です。

得点 学年 補足
90–100 5学年 非常に読みやすい。平均的な11才の生徒が容易に理解できる。
80–90 6学年 読みやすい。日常会話に使われる英語。
70–80 7学年 わりと読みやすい。
60–70 8、9学年 プレイン・イングリッシュ。平均的な13～15才の生徒が容易に理解できる。
50–60 10～12学年 わりと読みにくい。
30–50 大学生 読みにくい。
0–30 大学卒業 非常に読みにくい。最も理解できるのは大学卒業者。

訳註

学年はアメリカ合衆国の教育における学年です。7～9学年が日本における中学生程度、10〜12学年が高校生程度に相当します。プレイン・イングリッシュは主に政府の公式文書などに使われる、専門用語を避けて一般の人にわかりやすくした英語を指します。

TRAYリーダビリティツール（Chromeブラウザ拡張）など、Flesch-Kincaidを基準としてコンテンツをテストするツールはいくつかあります。コマンドラインからテストを実行したい場合には、私が作成したReadability Checker CLI――Node.js CLI（コマンドラインインターフェイス）があります。readability-checkerをインストールして、Webページを指定するだけです。

npm i readability-checker -g

readability http://your-site.com/about.html

以下が返されます。

  • Flesch Reading Easeの得点（100点満点、高いほど読みやすい）
  • 得点に関する補足
  • 長い単語の一覧（5音節以上）
  • 長い文章の一覧（36単語以上）

最近、アクセシビリティとインクルージョンにフォーカスした製品としてTenonのような品質保証ツールが登場しはじめました。開発者が開発者のために作ってきた、コードのメンテナンス性を中心とした品質保証という文化がある中で、これは非常に歓迎すべきことです。インデントが一貫していないなど、望ましくないコーディングスタイルでビルドが止まるというのは、よくあるやり方です。では、製品がユーザーを疎外する危険性がある、という理由でビルドが止まるとしたらどうでしょう？さらに良くなりますね。読みやすさが劣るという理由でビルドが失敗するようにすることを、避けるべき理由はないように思います。

ただし、この読みやすさはかなり主観的であることに注意してください。読みやすさを判定する公式は科学的に見えるかもしれませんが、Flesch-Kinkaidテストだけに頼るべきではありません。潜在的な問題をあぶりだす早期警戒システムとして利用した上で、実際のユーザーでもテストするようにしてください。

見出しとリンクテキスト

フロントエンドエンジニアとして働く人は、間違いなくモジュール設計の利点に慣れ親しんでいます。再利用可能なモジュール（コンポーネント）を作っておけば、より迅速に開発でき、十分にテストされたコードを利用でき、冗長性も排除できます。適切なモジュールには、多様なコンテキストでシームレスに役立つ、独立した目的があります。

インクルーシブデザインでは、テキストにある程度の自律性が求められる場合があります。例として以下の<h2>見出しを見てみましょう。

<h2>無料ですって？それならください！</h2>

前後の段落の文脈を見れば、見出しが何を指しているか推測することは間違いなく可能でしょう。つまり、前のセクションからこのセクションへ読み進めることで、何が無料なのかがわかるはずです。しかし、目次ではそういうわけにはいきません。本文から外れたところで見出しだけをリストにしているからです。何が無料なのでしょう？

ダイレクトで説明的な見出しは、あとに続く内容を明確にし、全体的な理解も助けます。言い換えれば、もったいぶったり、風変わりにしたりしても、ろくなことがないということです！さらに、スクリーンリーダーユーザーにとっては別の意味合いもあります。スクリーンリーダーは、動的に見出しをピックアップし、選択できるリストとして提供します。たとえばNVDAでは、Insert + F7キーで要素リストダイアログを開き、ページの見出しのリストを見ることができます。VoiceOverでは、ローター（Mac OS X上でCtrl + Option + U）を開いて、同じようにリストされた見出しを見ることができます。

訳註

NVDAのコマンド一覧表では、要素リストを開く操作は NVDA + F7 と表記されています。NVDAでは、NVDAの機能を使う際に押すキーを「NVDAキー」と呼んでおり、初期設定ではこれをInsertキーに割り当てていますが、設定で別のキーを使うようにすることもできます。

したがって、以下の見出しのほうがずっと便利でしょう（少なくともブログ記事のこのセクションが本当にお菓子のフラップジャックに関するものであれば。そうでなければ誤解を招きます！）。

<h2>フラップジャックが無料ですって？ください！</h2>

不快なリンクラベル

リンクも、見出しと同様にNVDAの要素リストダイアログに表示されます。他のスクリーンリーダーにも同じような機能があります。見出しと同様に、リンクテキストもそれ自体で意味がわかるようにすべきです。以下のように、インラインリンクを連続で使う癖のあるブロガーを見たことがあるのではないでしょうか。

私のすばらしいアイデアには、これを支える多 く の裏づけがある。

キュートですが、コンテキストから取り出してみると、それぞれのリンクには完全に無意味なラベルがついています。順番に「多、リンク。く、リンク。の、リンク」と読まれても、有益な情報は何も得られません。加えて視覚的にも、リンクが分割されていることがわからない可能性があります。アンダーラインが途切れていることに、ユーザーは気がつくでしょうか。実際にリンクアンダーラインを提供していないケースでは、さらにひどいことになります。

他人の著作物を引用するのであれば、バイネームでこれに言及するのが礼儀です。以下の例のリンクはすべて、実際のリンク先の場所を示しており、読点――と、「そして」という単語――によってそれぞれが分割されています。

私のすばらしいアイデアには、これを支える多くの裏づけがある。たとえば、John Thoughtleader氏の「ヘイドンが正しい理由」、Jane Unicornfield氏の「ヘイドンを支持する」、そしてHarry Surname氏の「ヘイドンが語るとき、それはかぐわしい虹を吐くかのようだ」などだ。

ビデオ

HTMLのそもそもの基本機能は、単語の並びやかたまりを差別化し、明確化することです。書き手として、私はいまでもこの機能に一番ワクワクさせられます。しかし、すべての人が自信あふれる書き手、あるいは熱心な読者というわけではないことも理解しています。画像、サウンド、ビデオをブログ記事に加えることで、コンテンツに特色や多様性が加わるだけでなく、さまざまな能力や好みを持つ人たちや多様な環境にいる人たちに、より多くの選択肢を提供できます。

ビデオを例にとりましょう。何かの仕組みを伝えるとき、言葉で説明するよりも、見てもらう方がよい場合があります。何かを学ぶのであれば、単純でよりよい方法だと言えるでしょう。逆に、同時に何か他の物を見る必要があるときなど、見せられるのではなく、言葉で説明して欲しいときもあります。たとえば、ビジュアルデザインの作業をしながら、同時にカンファレンスのトークビデオの会話を聞き取りたいと思うことがあります。映像を見ながらかつトークも聞きたいけれど、バスに乗っていてヘッドフォンがなく、しかし他の乗客に耳障りな音は聞かせられない、という場合もあるでしょう。さらに、ビデオで人が話しているとき、音声に加えてキャプションも見ることができれば、ネイティブスピーカーでない人にも理解しやすくなります。しっかりとキャプションづけされたビデオは、まさにおあつらえ向きです。

キャプションがあれば、もちろん聴力に問題のある人たちの役に立ちます。しかし、彼らのためだけのものではないのです。キャプションを見ることができるということは、単純に「同じコンテンツを利用する別の方法」が増えるということです。より多くの人に使えるようにする、という観点だけで見ても、キャプションをつけるのは良い考えでしょう。しかし、キャプションはビデオそのものがダウンロードできるか、またはストリーミング再生できる場合にしか利用できません。したがって、会話を含むビデオには常にトランスクリプトをつけるべきです。

ビデオブログの記事ではビデオを目立つように表示すべきですが、その下にビデオのトランスクリプトを提供した方が良いでしょう。以下は、ビデオを投稿するときのアドバイスです。

プレイヤー

ビデオプレイヤーがキーボードおよびスクリーンリーダーでアクセスできることを確認してください。採用に値するプレイヤーかどうかを見極める手っ取り早い方法は、キーボードを使ってプレイヤー内ににタブ移動してみることです。フォーカスがプレイヤー全体を飛ばしてしまったら、その時点でゲームオーバーです。撤退してください。そして別のプレイヤーを探してください。

キーボードアクセシビリティのワースト2は、タブ移動できて使用できるけれども、視覚的なフィードバックが一切ないというものです。3章「パラグラフ」の「フォーカスを示す」を思い出してください。簡単にいうと、フォーカスを消すスタイルはダメです。とはいえ、そのプレイヤーを構成するコントロールがHTMLとCSSで実装されていれば、コンテンツ側で独自のフォーカススタイルを提供することで、そのプレイヤーを救うことができるかもしれません。ただし、スクリーンリーダーでもアクセスできることが条件です。

スクリーンリーダーによるテストを始めてみるときは、以下をおすすめします。

  • Windowsの場合、無償のスクリーンリーダーNVDAをインストールする
  • Firefoxだけを立ち上げて実行する（NVDAとの互換性がもっとも高いため）
  • Deque Systemsの、このすばらしいビデオを見る。
  • WebAIMで、NVDAキーボードコマンドのリストを参照する。
  • 目が見えているユーザーによるスクリーンリーダーの体験は、視覚障害者の体験とは劇的に異なる場合があることに注意する。ひと口に視覚障害者といっても、使い方は人によって大きく異ります（私が行ったスクリーンリーダーの使い方調査の回答を見ると、その多様性がわかります）。

Macユーザーの場合は、代わりにCmd + F5を押して、OSに組み込まれているVoiceOverスクリーンリーダーをアクティブにし、チュートリアルに従ってください。実際のユーザーの体験に近づけるため、VoiceOverを使う際、ブラウザはSafariを使用してください。

幸いなことに、ビデオプレイヤーなどのインタラクティブなウィジェットをテストするのに、NVDAやその他のリーダーで提供されている膨大なキーボードコマンドに関する実用的な知識は必要ありません。キーボードが使えるかどうかをテストするときは、単にコントロールにタブ移動して、音声のフィードバックを聞いてください。たとえば、再生ボタンにフォーカスを合わせると「再生ボタン」（あるいは似たような別の言葉）、ビデオがすでに再生中の場合は「一時停止ボタン」、といった具合に読み上げられます。もちろん、このボタンが実際に、押すと再生が始まる状態から、押すと一時停止する状態に切り替わるかどうかも確認しなくてはなりません。スクリーンリーダーが実行されているかどうかに関わらず、SpaceかEnterキーを押せばボタンが動作するはずです。

クローズドキャプション

クローズドキャプションは、ビデオとは別のファイルに書かれ、ビデオに関連づけられたキャプション（字幕）です。ビデオ自体に字幕が埋め込まれているわけではないため、ユーザーは字幕のオン・オフを切り替えることができます。手動でHTML5の<video>にキャプションファイルを追加したい場合、<track>要素のsrc属性でWebVTTファイルを指定します。しかし、クローズドキャプションを設定する最も簡単な（と私が考えている）方法は、YouTubeの直感的なGUIを使用することです。Googleが、その使い方を説明したヘルプページを提供しています。

キャプションを書くときは、従うべき慣習があります。最も重要なのは、以下の点を認識できるようにすることです。

  • 話し手の変更
  • 環境音／状況を示す音

話し手の特定と効果音

ビデオにサイモン（Simon）とルパート（Rupert）という2人の男性が登場するとしましょう。2人はパブで席に座っています。サイモンが、ユウレイグモは世界で最も強い毒を持っていると話し、会話を始めます。続けて彼は、とはいえそのクモの歯は人間の肌を貫通できないため害はない、と説明します。

歯というのはもちろん毒牙のことを言っています。歯があるクモを想像してみてください！

以下の図では、大文字で名前を書くことによって、話し手を特定することにしました。話し手を区別する方法はこれだけではありませんが、どんな決まりを使う場合でも、必ず一貫したものにしてください。2番めのキャプションでは話し手が変わっていないため、名前が不要であることに注目してください。

ルパートは、この話が単なるインターネット上の噂であると知っていて、彼の間違った主張を正すため、サイモンの話をさえぎります。ルパートは新しい話し手であるため、話し手が彼になったことを認識できるようにしなければなりません。先ほどと同じように名前を表記します。

ルパートが話し終わる前に、バーテンダー（barman）が（フレームの外で）グラスを落としたため、2人の話はさえぎられます。多くの場合、このような物音の説明は、大文字で表記されるとともに [と]（角かっこ）ではさまれます。バーテンダーは、すぐに謝ります。大なり記号を先頭につけることで、彼がフレーム外にいることを示しています。

BBCは広範なガイドラインを提供しており、話し手、アナウンス、音、音楽などを示すさまざまな方法について触れています。

トランスクリプト

トランスクリプトは、ビデオ記事の主要なテキストコンテンツとなります。先にトランスクリプトを書き、それをキャプションに転用する方が楽な場合もあります。どちらにしても、トランスクリプトはキャプションをリニアライズした形をとります。ここで行った唯一の置き換えは、視覚的なコンテキストがない状態で明確に伝えるために、バーテンダーがフレームの外側にいると示したことです。

SIMON: 「ユウレイグモが世界で一番強い毒を持つ生き物だって知っていた？だけど実際は、歯が弱すぎて人は噛めないんだ」

RUPERT: 「ええと、本当は…」

[グラスの割れる音]

BARMAN（フレーム外）：「失礼しました！」

フローシステムの確立

インクルーシブWebデザインの目的は、ユーザーに堅牢でアクセシブルな体験を提供することだけではありません。サイトオーナーや編集者のコンテンツへの貢献を後押しすることでもあるのです。

グリッドシステムと同じく、フローシステムはCSSでレイアウトを管理します。グリッドシステムとは異なり、フローシステムでは、見出し、段落、リスト、画像、ビデオ、といった次々現れるフロー要素どうしの関係性を、ひとつのカラムだけで表現します。優れたフローシステムは、要素の順番や組み合わせがどのようなものであっても、要素の並び方が規則的で、十分な間隔があり、読みやすいようにしてくれます。これにより、編集者には、ただひたすら書くという自由が与えられるのです。

訳註

HTML5におけるフロー要素の分類は非常に幅広く、em要素などのフレーズ要素も含まれていますが、著者は主に <p>、<div> など (HTML4のブロック要素) をイメージしています。前著「コーディングWebアクセシビリティ」をお持ちの方は、「4-1 分割する」の章をご覧ください。

参考

  • HTML Living Standard - 3.2.5.2.2 Flow content

すべてはline-heightで分割できる

厳密なバーティカルリズムを確立して運用するのは大変なこととして知られていますが、ひとまずは段落のline-heightを縦の間隔の基準として使用するのが良いでしょう。つまり、段落のline-heightが1.5の場合、縦の空白の1単位を1.5remにします。

編集者のWYSIWYG（what you see is what you get：最終的に出力されるスタイルのまま編集できる）やMarkdownエディタによって出力されるフロー要素にもこの規則を強制するために、少数の例外を除いて、この基本の間隔を全体に適用する必要があります。

main * + * {
    margin-top: 1.5rem;
}

li, dt, dd, br, th, td {
    margin-top: 0;
}

最初の行でowlセレクタが使用されていることに注目してください。これにより、連続する要素の間だけに上マージンがつくようになります。それぞれの要素に直接マージンを適用してしまうと、コンテナのパディングとダブってしまう危険があります。

.container {
    padding: 1.5rem;
}

.container p {
    margin-top: 1.5rem;
}

/* .container上部の見た目上のスペースは3remになります */

グルーピングと分離

ここまでのコードサンプルでは、すべてのフロー要素は同じ1.5emのマージンで区切られていますが、例外はいくつかあります。リスト項目（<li>）にはマージンがまったくなく、結果としてグループ化されているように見えます。そして、そうであるべきです。読者の理解を助けるために、関連する要素は視覚的に近接させる必要があります。

ブログ内のコンテンツのサブセクションでも、リスト項目と同様の視覚的な近接効果を得ることができます。見出しのマージンの単位をremからemに変更すると、見出しのフォントサイズと相対的なマージンになります。見出しテキストは本文テキストよりも大きいため、見出しは下に続くコンテンツの方に近づいて見えるようになります。

* + h2,
* + h3 {
    /* 今度はremではなくem */
    margin-top: 1.5em;
}

引用やイラストなどの補足的なコンテンツを本文テキストから離すには、以下のように<p>以外のすべてのマージンを増やします。

main * + *:not(p) {
    margin: 3rem 0;
}

適切なバーティカルリズムを維持するために、段落の行の高さの倍数を指定しています（1.5 × 2 = 3）。すぐ下の要素には標準の1.5remの上マージンがついていますが、これは上書きされ、結果として、該当する要素の上下両方のマージンが指定どおりの3remとなることに注目してください。これはマージンの相殺ルールのおかげです。 編集者は、<blockquote>、<figure>、<audio>、その他ありとあらゆる要素を挿入することがありますが、このルールはどんな要素が来ても常に間隔を空けて、文章の小休止を伝えてくれるのです。


CSSでもっとも包括的なものは全称セレクタ (*) です。うまく使えば、コンテンツの相互関係を包括的に管理できます。

防御的なコーディングについての注意

WYSIWYGという言葉が少し誤解されているのをよく目にします。WYSIWYGエディタでは、ほぼ必ず、意図していない見えないゴミがソース内にたくさん残ります。

<p></p>
<p></p>
<p></p>
<p></p>
<!-- 際限なく続く -->

環境によっては、これがコンテンツの間隔やバーティカルリズムに影響することがあります。ここでは防御的なコーディングを採用して、フローから空の要素を取り除きましょう。display:none;の宣言は、マージンも含めて要素を削除します。

main :empty {
    display: none;
}

訳註

:empty擬似クラスは、子ノードを持たない要素にマッチします。空要素にもマッチするため、このセレクタをそのまま使うとimg、br、hrなども全て削除されてしまいます。imgが消えては困る場合は、main :not(img):emptyのようにしてimgを除外するか、main p:emptyのようにして空のpだけにマッチさせると良いでしょう。

一般論としては、MarkdownやTextileのような構文を使用して、セマンティックで十分に構造化されたコンテンツを作成できるなら、それがベストではあります。これらはシンプルな構文でありながら、見出しやリスト項目などのセマンティックなHTML要素に対応しています。ほとんどのエディタ（本書の執筆に使用したPenflipもそうです）では、必要に応じて生のHTMLをそのまま書くこともできます。編集者がHTMLに詳しければ、複雑なコンテンツも追加できるということです。

とはいえ、WYSIWYGエディタを構造化モードに切り替えて、レガシーなWYSIWYGベースのシステムにパッチを当てるように編集スタッフを教育するのは大変なことです。その代わりに防御的な戦略をとれば、頭痛も和らぐでしょう。コンテンツの入力時にレイアウトのバグを修正するのに費やす時間は無駄であり、編集作業をストレスのたまるとっつきにくい体験にしてしまう原因にもなります。

まとめ

ここまで、ブログ記事のパターンを構想しながら、インクルーシブデザインの最も重要な側面である構造について説明してきました。アクセシブルなランドマークと、音として聞こえるセクション構造を組み立てることで、コンテンツ――ブログ記事やその他のもの――は、ナビゲートしやすいものになり、多様なユーザーやパーサーの相互運用性が向上します。それを補強するために、適切に文章を書き、コンテキストに頼らなくても理解できるラベルを使うようにしました。その次に、ビデオなどのメディアを追加して、この堅牢な構造を補完する方法も説明しました。

本文テキスト、見出し、メディアといった要素をひとつの型にはめるのではなく、適切な構造を保ちながらも数の制約なしに提供できるように、堅牢で包括的なCSSフローシステムを採用しました。これにより、編集者は視覚的なデザインを破綻させる心配をせずに済み、自由に書くことができるようになるでしょう。

避けるべきこと

  • ランドマークや見出しなど、ブロックスキップに必要となるナビゲート可能な構造の手がかりを用意しない。
  • 見出しやリンクに風変わりなラベルやもったいぶったラベル、または部分的なラベルをつける。
  • CSSのマージンの宣言を、コンテキストの変化に影響されやすく、寛容性がないものにする。
  • 静的コンテンツのサイトを、クライアント側のJavaScriptに依存したつくりにする。
